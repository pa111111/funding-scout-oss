Ручной флоу + $5k на связку (см. execution.md) → продукт не исполняет, но **обязан выводить пользователю чек-лист с правилами**, которые опытные арбитражники соблюдают по умолчанию. Источник правил: стрим Дарис + Никита, расшифровка обработана 2026-05-02.

## Открытие позиции

1. **Изолированная маржа на обеих ногах. Mandatory.** Кросс-маржа = главная причина потерь 50%+ на flash-crash 10.10.2025. Если на DEX нет изолированной маржи (некоторые из новых) → использовать **субаккаунт под каждую позицию**.
2. **Один аккаунт = одна позиция.** Не открывать несколько позиций на одном аккаунте даже на изолированной — путаница в учёте, риск кросс-влияния.
3. **Sizing — equal contracts (количество токенов), не equal $.** Из-за price spread между биржами один и тот же $-объём даст разное число контрактов → дельта не нейтральна. Алгоритм: на бирже A открываем нужный размер, копируем абсолютное число токенов, на бирже B вводим то же число.
4. **Гибридное исполнение лимит/маркет.** Для DEX без комиссии (Lighter, Variational) — заходим маркетом (быстро, не теряем на спреде стакана дороже комиссии). Для DEX с комиссией (Hyperliquid: maker 0.015% vs taker 0.045%, разница 3×) — заходим лимиткой. Это режет round-trip cost на 30–50%.
5. **Плечо: 2–2.5x — 3x.** Не 1x (неэффективный капитал — лучше тогда чистый спот без фандинга). Не 5x+ (узкий рейндж до ликвидации → частые перезаходы → комиссии съедают APR).
6. **Spread-check перед входом.** Если цена актива на venue, где мы должны лонговать, выше чем на venue для шорта → конвергенция работает против нас. Не входить.

## Удержание

1. **TP и SL ставим близко к цене ликвидации (~10% запас), но не вплотную.** Слишком близкий стоп = частые срабатывания на нормальной волатильности. Слишком далёкий = риск долёта до ликвидации с биржевым штрафом ~1%.
2. **TP и SL только market, не limit.** Лимитные стопы регулярно пролетают (Mona на 32%, многие позиции 10.10.2025 на 30–40%). Жадность ради лучшей цены закрытия не оправдана.
3. **Накопленный фандинг сдвигает цену ликвидации в выгодную сторону → переставлять стопы.** За несколько дней фарма стоп может оказаться слишком далеко от текущей ликвидации (Никита потерял на этом часть денег на Mona). Проверка стопов раз в день обязательна.
4. **Routine: утро 30 мин + вечер 30–60 мин.** Утром — проверить, не сломалось ли что-то за ночь. Вечером — pf equity всех бирж, сравнение с балансом за вчера, ребалансировка стопов и ребалансировка капитала между ногами при сильных движениях.

## Закрытие

1. **Выходим, когда APR падает ниже стоимости перезахода** (round-trip cost / реальная APR per period).
2. **Никаких "подожду — сейчас вернётся".** Если связка ушла — закрываем как есть. Удержание одной ноги в ожидании = ставленый трейд, выход за рамки стратегии. Это ровно тот сценарий, через который дельта-нейтральщики становятся лудоманами.
3. **Закрытие тоже зеркальное по числу токенов**, не по %. Тот же принцип, что при входе.

## Учёт PnL

1. Считать **только portfolio equity на каждой бирже** (сухие балансы). Не складывать комиссии, фандинг и нереализованный PnL отдельно — это уже включено в equity.
2. Делать **скриншоты балансов вечером каждого дня**, сравнивать с предыдущим днём. Никита автоматизирует через GPT-промпт, который собирает таблицу из скриншотов — это паттерн, который полезен для будущего log-модуля funding-scout.
3. Эта же таблица — основа для post-mortem и backtester ground truth.

## Risk-метрики (не gates, а показатели — см. парадигму в concept.md)

В карточке связки рядом с базовым EV отображаются как **отдельные оси**, без штрафа в EV и без автоматической фильтрации:
- Поддержка изолированной маржи на venues / необходимость субаккаунта.
- Минимальное плечо для разумной APR (если 8x+ — флаг "хрупкая по плечу", но связку не отбрасываем).
- Pre-market флаг (если да — 🔴 badge, но связка в листе).
- β-эластичность, σ часовая (см. risk.md) — числа + цвет.
- Симулированный flash-crash 10.10.2025 — конкретная цифра потери (например "−34% used capital при market-stops"), пользователь сам решает, в его envelope или нет.

Hard-filter по любой из этих метрик существует только как **saved user filter profile** — пользователь сам выбирает, какие связки скрыть из своего листа. По умолчанию продукт показывает всё, оставляя выбор за оператором.
