Решения от 2026-05-02: продукт внутренний, исполнение ручное, объём на одну связку до $5k. Это меняет модель исполнения и вводит блок "execution friction".

## Базовый принцип

Связка с funding APR 200% и spread convergence 1.5%, но требующая бриджа на $5k за $50 fee + 30 мин ожидания + 0.3% slippage свопа = реально тратит ~30% ожидаемого EV ещё до открытия позиции. Топ-list, который игнорирует это, врёт пользователю.

Поэтому каждая связка обогащается двумя полями:
- **Required venues** — на каких DEX нужны деньги для каждой ноги, в каком объёме
- **Friction tax %** — стоимость мобилизации капитала с текущего его расположения, выраженная как доля от ожидаемого EV

## Capital state (новый блок архитектуры)

Минимально достаточно ручной таблицы вида:
```
{ venue: "Hyperliquid",  balance_usd: 8000 },
{ venue: "Pacifica",     balance_usd: 5000 },
{ venue: "Lighter",      balance_usd: 0 },
...
```
Обновляется руками (раз в день/после трейдов) или через DEX API при наличии read-only доступа (предпочтительно, если возможно). Точные цифры не критичны — важна категория ("есть капитал / нет капитала / частично").

В roadmap: автоматическое обновление через подписные DEX-API.

## Friction calculator

Для каждой требуемой ноги, у которой не хватает капитала на venue:
- **Bridge fee** — фиксированная стоимость моста между сетями (если ноги на разных L1/L2)
- **Bridge time** — минуты, в течение которых spread может убежать (умножаем на текущую σ актива → "expected adverse drift")
- **Swap slippage** — если нужно сконвертировать актив (USDC → нативный коллатерал биржи)
- **Gas** — сумма по всем on-chain транзакциям

Сумма этих компонент / ожидаемый абсолютный профит = friction tax %.

## Ранжирование под ручной флоу

UI делит кандидатов на две группы:
- 🟢 **Доступно сейчас** (friction tax = 0) — капитал уже на всех нужных venues. Показываем первыми, сортируем по EV.
- 🟡 **Требует переброски** (friction tax > 0) — показываем ниже, явно цифру friction и net-EV после неё. Многие связки тут окажутся бессмысленными после вычета friction — это нормально.

Это автоматически отсекает класс ложно-привлекательных связок: "топ-APR, но с другого конца света по капиталу".

## Что выпадает из MVP под ручной флоу

- **Sniping (тип 4)** — окно минуты, руками не успеть. Не детектим.
- **Order routing / smart execution** — пользователь сам открывает руками.
- **Private key management** — нет в продукте вообще.
- **Auto-rebalance капитала по DEX** — слишком сложно при много-DEX и разных сетях.

## Когда автоматизация снова в обсуждении

Триггеры пересмотреть ручной флоу:
- Объём одной связки вырос с $5k до $50k+ (тогда даже small market impact окупает разработку executor).
- В таксономии появляется class связок с окном <5 минут, который нельзя ловить руками, но он стабильно прибыльный по бэктесту.
- Стало понятно, что сам пользователь тратит >2 часов в день на ручное выполнение алертов — тогда стоимость разработки executor окупается экономией времени.
